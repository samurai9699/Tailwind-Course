Git and CI/CD are the two most foundational skills at any software job ‚Äî and they're almost never taught properly. Most tutorials either dump too much on you at once, or skip the "why" entirely and just list commands. Neither approach sticks.

This course works differently. We start with mental models ‚Äî real-world analogies that make the concepts click. Then we build up layer by layer, from the very first command to the patterns senior engineers use daily. By the end, nothing your colleagues say in standup will confuse you.

Git ‚Äî Version Control

02

What is Git?

Git is a version control system. It tracks every change you make to files over time, lets you go back to any previous state, and lets multiple people work on the same codebase without destroying each other's work.

üì∏

The Photo Album Analogy

Imagine your project is a photo of a room. Every time you make a meaningful set of changes, you take a new photo (a "commit"). Git stores all those photos in order. You can flip back to any photo from the past. You can also create a parallel photo series (a "branch") where you experiment with redecorating ‚Äî and merge it back into the main series when you're happy.

Git vs GitHub ‚Äî what's the difference?

This confuses almost everyone starting out. They are completely different things:

üß∞

Git

A program installed on your computer. It's the engine. It does the actual tracking, branching, and merging. Works completely offline.

‚òÅÔ∏è

GitHub

A website that hosts your Git repositories online. It's like Dropbox for code ‚Äî but with extra features for teams like Pull Requests, Issues, and Actions (CI/CD).

ü¶ä

GitLab

Another code hosting website, similar to GitHub. Many companies self-host it. Has its own built-in CI/CD system (GitLab CI).

ü™£

Bitbucket

Another hosting option, popular in corporate environments. Common with Atlassian tools like Jira and Confluence.

üí° The key insight Git = the tool on your laptop. GitHub/GitLab/Bitbucket = the cloud home for your code. You can use Git without GitHub, but you can't use GitHub without Git.

Setting Up Git

Before anything else, let's get Git installed and configured on your machine.

Installation

terminal ‚Äî install git
# Mac (using Homebrew) brew install git # Ubuntu / Debian Linux sudo apt-get install git # Windows # Download from: https://git-scm.com/download/win # Or via winget: winget install Git.Git # Verify installation git --version git version 2.43.0

First-time configuration

Git needs to know who you are. This info gets attached to every commit you make. Run these once ‚Äî ever.

terminal ‚Äî configure git
# Set your name (shows up in commit history) git config --global user.name "Your Name" # Set your email (should match your GitHub email) git config --global user.email "you@email.com" # Set your preferred editor (VS Code shown here) git config --global core.editor "code --wait" # Set the default branch name to "main" (modern standard) git config --global init.defaultBranch main # Verify your config git config --list
üí° The --global flag This saves the setting to your user profile, so it applies to ALL Git repos on your computer. You can override it per-project by running the same commands without --global inside a specific folder.
04

Your First Repository

A repository (repo) is just a folder that Git is tracking. Let's create one and understand the core Git loop.

The Three Areas of Git

This is the most important concept in Git. Almost all confusion comes from not understanding these three areas:

üìÅ
Working Directory ‚Äî your actual files on disk. Where you edit code. Git watches this but doesn't track changes automatically.
üé≠
Staging Area (Index) ‚Äî a "waiting room" for changes you're about to commit. You choose exactly what goes into the next snapshot.
üì∏
Repository (History) ‚Äî the permanent record of commits. Once something is committed, it's saved in history forever (unless you intentionally remove it).

Creating your first repo

terminal ‚Äî your first repo
# Create a new folder and enter it mkdir my-project cd my-project # Initialize Git ‚Äî this creates a hidden .git folder git init Initialized empty Git repository in /my-project/.git/ # Create a file echo "# My Project" > README.md # Check the status ‚Äî what does Git see? git status On branch main Untracked files: (use "git add ..." to include in what will be committed) README.md

Git sees the file but it's "untracked" ‚Äî Git isn't watching it yet. Let's add it to the staging area, then commit it.

terminal ‚Äî stage and commit
# Stage the file (move it to the staging area) git add README.md # Or stage EVERYTHING in the folder git add . # Check status again ‚Äî notice it changed git status Changes to be committed: (use "git rm --cached ..." to unstage) new file: README.md # Commit with a message git commit -m "Initial commit: add README" [main (root-commit) a3f2c1d] Initial commit: add README 1 file changed, 1 insertion(+) # See the commit history git log commit a3f2c1d... Author: Your Name <you@email.com> Date: Mon Feb 23 2025 Initial commit: add README # Cleaner one-line log view git log --oneline a3f2c1d Initial commit: add README

Writing good commit messages

Commit messages are letters to your future self (and teammates). Bad messages make debugging a nightmare. Good messages are priceless.

‚ùå Bad commit messages (do not do this) "fix" / "stuff" / "asdfgh" / "changes" / "working now" / "lol"
‚úÖ Good commit message format Start with a verb in present tense. Short subject line (under 72 chars). Explain the WHY if it's not obvious. Examples: "Add user authentication endpoint" ¬∑ "Fix race condition in cart checkout" ¬∑ "Remove deprecated payment API calls"

‚úî Checkpoint ‚Äî you should now understand

  • The difference between Git and GitHub
  • What a repository, commit, and staging area are
  • The flow: edit ‚Üí git add ‚Üí git commit
  • How to read git status and git log
05

Branching ‚Äî Working in Parallel

Branching is the feature that makes Git genuinely powerful. A branch is simply a separate line of development ‚Äî a copy of your code where you can make changes without affecting the "main" copy until you're ready.

üåø

The Tree Analogy

The main branch is your tree trunk ‚Äî the stable, production-ready code. When you work on a new feature, you grow a new branch off the trunk. You can change that branch freely without affecting the trunk. When the feature is done and tested, you "merge" the branch back into the trunk.

Creating and switching branches

terminal ‚Äî branches
# See all branches (* = current branch) git branch * main # Create a new branch git branch feature/user-login # Switch to it git checkout feature/user-login # Or do both at once (modern, preferred way) git checkout -b feature/user-login Switched to a new branch 'feature/user-login' # Even newer syntax (Git 2.23+) git switch -c feature/user-login

Now make some changes and commit them on this branch. Those commits only exist on this branch ‚Äî main is untouched.

Merging a branch

terminal ‚Äî merging
# First, go back to main git checkout main # Merge your feature branch into main git merge feature/user-login Updating a3f2c1d..7b4e9f2 Fast-forward login.js | 24 +++++++++++++++++++++++++ 1 file changed, 24 insertions(+) # Delete the branch after merging (cleanup) git branch -d feature/user-login Deleted branch feature/user-login (was 7b4e9f2).

Branch naming conventions

Good teams use consistent naming. The most common patterns you'll see in real jobs:

branch naming patterns
# New features feature/user-authentication feature/dark-mode feat/payment-integration # Bug fixes fix/login-redirect-bug bugfix/cart-price-calculation hotfix/critical-security-patch # Other types chore/update-dependencies docs/api-documentation refactor/cleanup-auth-module
06

Remote Repositories ‚Äî Working with GitHub

Until now, everything has been local ‚Äî just on your machine. Now let's push your code to GitHub so others can see it (and you can collaborate).

Key remote commands

terminal ‚Äî working with remotes
# Clone an existing repo from GitHub to your machine git clone https://github.com/username/repo-name.git # Link your local repo to a remote (after creating on GitHub) git remote add origin https://github.com/username/repo-name.git # See your remotes git remote -v origin https://github.com/username/repo-name.git (fetch) origin https://github.com/username/repo-name.git (push) # Push your commits to GitHub (-u sets the "upstream" for tracking) git push -u origin main # After first push, just do this git push # Get changes others have pushed git pull # Fetch without merging (safer ‚Äî see changes before applying) git fetch

Pull Requests (PRs) ‚Äî the team workflow

In real jobs, you almost never push directly to main. Instead, you push your feature branch and open a Pull Request. A PR is a formal request to merge your branch, with a code review process attached.

  1. Create a feature branch: git checkout -b feature/my-thing
  2. Make changes, commit them, push the branch: git push -u origin feature/my-thing
  3. Go to GitHub ‚Üí your repo ‚Üí "Compare & pull request"
  4. Write a description of what you changed and why
  5. Request reviews from teammates
  6. Address review comments, push more commits if needed
  7. Merge the PR once approved
  8. Delete the branch (GitHub offers a button for this)
‚ÑπÔ∏è Why PRs matter Pull Requests aren't bureaucracy ‚Äî they're how teams catch bugs before they reach production, share knowledge, and maintain code quality. Senior engineers spend significant time reviewing PRs. Getting good at writing and reviewing them is a huge career accelerator.

Keeping up with a busy team (fetch + rebase)

terminal ‚Äî staying in sync
# You're on your feature branch, main has moved ahead. # Option 1: merge main into your branch git checkout feature/my-thing git merge main # Option 2: rebase your branch on top of main (cleaner history) git fetch origin git rebase origin/main # Most common shorthand to update and sync git pull --rebase origin main

Fixing Mistakes ‚Äî Git is Your Safety Net

One of Git's superpowers is that almost nothing is truly permanent. Here's how to undo things at every stage.

Unstage a file

terminal ‚Äî fixing mistakes
# Oops, staged the wrong file git restore --staged wrong-file.js # Discard changes in working directory (CAREFUL - permanent) git restore wrong-file.js # Amend the last commit (change message or add missed file) git add forgotten-file.js git commit --amend -m "Better commit message" # ‚ö†Ô∏è Don't amend if you've already pushed to shared branch! # Undo last commit but KEEP the changes staged git reset --soft HEAD~1 # Undo last commit and UN-stage the changes (keep files) git reset --mixed HEAD~1 # Undo last commit and DISCARD all changes (nuclear option) git reset --hard HEAD~1 # Undo a commit safely (creates a new "reversal" commit) # Safe to use on shared branches! git revert abc1234

Stashing ‚Äî temporarily set work aside

You're mid-work on a feature, but someone needs you to urgently fix a bug on main. Stash saves your in-progress changes so you can come back to them.

terminal ‚Äî git stash
# Save current changes to a temporary "stash" git stash Saved working directory and index state WIP on main # Give your stash a name (helpful when you have multiple) git stash push -m "half-done login feature" # See all stashes git stash list stash@{0}: On feature/login: half-done login feature stash@{1}: WIP on main: quick fix attempt # Get your stash back (and remove it from stash list) git stash pop # Apply a specific stash without removing it git stash apply stash@{1}
‚ö†Ô∏è reset --hard is permanent git reset --hard deletes your uncommitted changes with no recovery. Use it only when you're absolutely sure. git revert is almost always the safer choice for anything you've already committed.
08

Merge Conflicts ‚Äî Don't Panic

A merge conflict happens when two branches change the same part of the same file in different ways, and Git doesn't know which version to keep. This is completely normal and expected in team work. Here's how to handle it calmly.

What a conflict looks like

conflict markers in a file
# When you open the conflicted file, you'll see this: function greetUser(name) { <<<<<<< HEAD return `Hello, ${name}! Welcome back.`; ======= return `Hey ${name}! Good to see you.`; >>>>>>> feature/friendlier-greeting } # HEAD = your current branch's version # The other side = the branch being merged in # The ======= line is the divider

Resolving the conflict

  1. Open the conflicted file in your editor. VS Code highlights conflicts nicely with "Accept Current", "Accept Incoming", "Accept Both" buttons.
  2. Edit the file to be what you actually want. Remove the conflict markers (<<<, ===, >>>).
  3. Save the file.
  4. Stage the resolved file: git add filename.js
  5. Complete the merge: git commit
üí° Pro tip: Use a merge tool VS Code has a built-in merge editor. You can also use tools like git mergetool. But honestly, for most conflicts, just edit the file manually ‚Äî it's the fastest way.

Avoiding conflicts

You can't avoid them forever, but you can minimize them: keep branches short-lived, pull from main frequently, communicate with teammates about which files you're working on, and keep functions small so two people rarely edit the exact same lines.

Advanced Git ‚Äî The Senior Toolkit

These commands are what separate someone who "knows Git" from someone who's truly comfortable with it. You'll hear senior engineers use these regularly.

Rebase ‚Äî rewriting history

Rebase takes your commits and replays them on top of another branch, as if you'd started your work from a different point. The result is a clean, linear history.

terminal ‚Äî rebase
# You're on feature/my-thing # main has had 3 new commits since you branched off # You want your commits to appear AFTER those 3 git rebase main Successfully rebased and updated refs/heads/feature/my-thing. # Interactive rebase: clean up your last N commits before a PR # Opens an editor where you can squash, reorder, rename commits git rebase -i HEAD~3 # In the editor that opens, you'll see: pick a1b2c3 Add login form pick d4e5f6 Fix typo in form pick g7h8i9 More login form tweaks # Commands: pick=keep, squash=combine with prev, reword=rename
‚ö†Ô∏è Golden rule of rebasing Never rebase a branch that other people are working on. Rebasing rewrites commit hashes, which will break everyone else's copy. Only rebase your own local/private branches.

Cherry-pick ‚Äî stealing a specific commit

terminal ‚Äî cherry-pick
# Take a specific commit from another branch and apply it here # Useful for hotfixes: "grab that fix from the feature branch" git cherry-pick a1b2c3d [main 8f7e6d5] Fix critical null pointer in checkout Date: Mon Feb 23 2025 1 file changed, 3 insertions(+), 1 deletion(-)

Git log ‚Äî advanced history reading

terminal ‚Äî power log commands
# Visual branch graph in terminal git log --oneline --graph --all # Who changed this file and when? git log --oneline -- src/login.js # Who wrote each line of a file? (blame is not an accusation!) git blame src/login.js # What changed between two commits or branches? git diff main feature/my-thing # Search for when a word was added or removed git log -S "payment_processor" --oneline # Find which commit introduced a bug (binary search) git bisect start git bisect bad HEAD git bisect good v1.0.0 # Git checks out commits halfway through, you test and mark good/bad

Tags ‚Äî marking releases

terminal ‚Äî tags
# Create a tag at the current commit git tag v1.0.0 # Annotated tag (preferred ‚Äî includes message, author, date) git tag -a v1.0.0 -m "Release version 1.0.0" # Push tags to GitHub (they don't push automatically) git push origin v1.0.0 git push origin --tags # List all tags git tag

Real Workflows ‚Äî How Teams Actually Work

Knowing Git commands is one thing. Knowing how teams use Git together is what makes you productive on day one of a new job.

Feature Branch Workflow (most common)

This is what most startups and small-to-medium teams use. Main is always stable. All work happens in branches.

daily workflow
# Start your day: update main git checkout main git pull # Create your branch for today's task git checkout -b feature/add-dark-mode # Work, work, work... commit regularly git add . git commit -m "Add dark mode toggle to settings page" # Before pushing, rebase to include latest main changes git fetch origin git rebase origin/main # Push your branch git push -u origin feature/add-dark-mode # Go to GitHub ‚Üí open Pull Request # Get reviews ‚Üí address feedback ‚Üí merge ‚Üí delete branch

GitFlow (larger teams/enterprise)

GitFlow adds structure with multiple long-lived branches. You'll see this at companies with scheduled release cycles.

üèóÔ∏è

main

Production-ready code only. Every commit is a release. Tagged with version numbers.

üîß

develop

Integration branch. All features merge here first before going to main.

‚ú®

feature/*

Branch off develop. One per feature. Merge back into develop when done.

üöë

hotfix/*

Branches off main for urgent production fixes. Merged into both main and develop.

Trunk-Based Development (modern/fast teams)

Everyone commits small changes directly to main (or very short-lived branches). Requires excellent automated testing and feature flags. Used by Google, Facebook, and most high-velocity engineering orgs.

Part 02

CI/CD ‚Äî Automated Pipelines

Learn to build automated systems that test, build, and deploy your code every time you push ‚Äî the backbone of modern software teams.

12

What is CI/CD?

CI/CD stands for Continuous Integration / Continuous Delivery (or Deployment). It's the practice of automatically testing and deploying your code every time a change is pushed ‚Äî instead of doing it manually and infrequently.

üè≠

The Assembly Line Analogy

Before assembly lines, cars were built by hand, slowly, with lots of errors. CI/CD is the assembly line for software. Every time you push code, it automatically runs through quality checks (tests), assembly (build), and delivery (deploy). No human needs to manually trigger it. If something is wrong, the line stops and you get notified immediately.

CI vs CD ‚Äî what's the difference?

üîÑ

Continuous Integration (CI)

Automatically run tests and checks every time someone pushes code. Goal: detect problems early, before they merge to main.

üì¶

Continuous Delivery (CD)

Automatically prepare a deployable build after CI passes. The code is always ready to deploy ‚Äî but a human still clicks "deploy".

üöÄ

Continuous Deployment

Like Delivery but fully automatic ‚Äî every passing change automatically goes to production with zero human intervention.

üõ°Ô∏è

The Combined Value

Together they mean: you can push code 50 times a day and always know it works. Bugs get caught in minutes, not days.

Why does this matter for you?

At your first job, you will almost certainly encounter a CI/CD pipeline on your first day. When you open a PR, tests will automatically run. If they fail, you'll need to understand why. Knowing how pipelines work lets you fix them, improve them, and look extremely competent early on.

Core Concepts ‚Äî The Vocabulary

CI/CD tools have their own jargon. Once you know these terms, you can understand any pipeline ‚Äî regardless of which tool (GitHub Actions, GitLab CI, Jenkins, CircleCI) it uses.

üîÅ

Pipeline

The entire automated process from code push to deployment. A pipeline contains multiple stages that run in order.

üìã

Stage / Job

A distinct phase of the pipeline (e.g., "test", "build", "deploy"). Each stage can have multiple steps. Jobs in the same stage may run in parallel.

‚ö°

Runner / Agent

The server that actually executes your pipeline. GitHub provides cloud runners; companies sometimes use self-hosted runners.

üéØ

Trigger

What starts the pipeline. Common triggers: push to a branch, pull request opened, a schedule (cron), or manual trigger.

üì§

Artifact

A file produced during the pipeline (a compiled binary, test report, Docker image) that can be passed between stages or downloaded.

üîê

Secret / Environment Variable

Sensitive values (API keys, passwords) stored securely and injected into the pipeline at runtime. Never hardcoded in YAML files.

What a pipeline looks like visually

üß™ Test

‚Üí
üî® Build

‚Üí

If any stage fails, the pipeline stops and notifies you. Nothing broken reaches production.

Your First Pipeline ‚Äî GitHub Actions

We'll use GitHub Actions ‚Äî it's built into GitHub, free for public repos, and extremely popular in 2025. The concepts transfer directly to GitLab CI and others.

GitHub Actions pipelines are defined in YAML files stored in your repo at .github/workflows/. When you push code, GitHub reads these files and runs them.

Your first workflow file

.github/workflows/main.yml
# The name of this workflow (shows in GitHub UI) name: CI Pipeline # When should this run? on: push: branches: [ main, develop ] pull_request: branches: [ main ] # The actual work jobs: # "test" is the job name ‚Äî you choose it test: # Which type of machine to run on runs-on: ubuntu-latest # The steps to execute, in order steps: # Step 1: Get your code - name: Checkout code uses: actions/checkout@v4 # Step 2: Set up Node.js - name: Set up Node.js uses: actions/setup-node@v4 with: node-version: '20' # Step 3: Install dependencies - name: Install dependencies run: npm install # Step 4: Run tests - name: Run tests run: npm test

That's it. Push this file to your repo and GitHub will automatically run it. You'll see results in the "Actions" tab.

Understanding the YAML structure

üìã

name:

The display name for the workflow. Pick something descriptive.

üéØ

on:

Defines triggers. You can trigger on push, pull_request, schedule, manual (workflow_dispatch), and more.

üíº

jobs:

Groups of steps. Jobs run in parallel by default. Add "needs: [other-job]" to sequence them.

üëü

steps:

Individual tasks within a job. Can use pre-built "actions" (uses:) or run shell commands (run:).

‚ÑπÔ∏è What are "actions"? The uses: actions/checkout@v4 line uses a pre-built, reusable piece of code from GitHub's marketplace. Think of them as plugins. There are thousands ‚Äî for setting up languages, deploying to AWS, sending Slack notifications, and almost anything else you need.

Testing in CI ‚Äî Your Quality Gate

The most valuable thing CI does is run your tests automatically and block bad code from merging. Here's how to set this up properly.

Running tests for different languages

.github/workflows/test.yml ‚Äî multi-language examples
## NODE.JS / JAVASCRIPT - name: Run Node tests run: npm test ## PYTHON - name: Set up Python uses: actions/setup-python@v5 with: python-version: '3.12' - name: Install dependencies run: pip install -r requirements.txt - name: Run pytest run: pytest --tb=short ## JAVA / GRADLE - name: Set up JDK uses: actions/setup-java@v4 with: java-version: '21' distribution: 'temurin' - name: Run tests run: ./gradlew test

Test reports and coverage

.github/workflows/test-with-coverage.yml
jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' - run: npm ci # Run tests with coverage report - name: Run tests with coverage run: npm test -- --coverage # Upload coverage report as an artifact - name: Upload coverage report uses: actions/upload-artifact@v4 if: always() # upload even if tests fail with: name: coverage-report path: coverage/

Linting ‚Äî checking code style

Most teams also run a linter in CI. This enforces consistent code style so no one has to argue about formatting in code reviews.

adding a lint job
jobs: lint: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 - run: npm ci - name: Run ESLint run: npm run lint test: runs-on: ubuntu-latest needs: lint # only run tests if lint passes steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 - run: npm ci - run: npm test

Build & Artifacts

After tests pass, you often need to build your application ‚Äî compile TypeScript, bundle a React app, package a JAR file. The output is an artifact that gets deployed.

Build and cache dependencies

.github/workflows/build.yml
jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' # caches node_modules ‚Äî MUCH faster - run: npm ci # ci is faster + stricter than npm install - name: Build application run: npm run build # Save the built files for the deploy job - name: Upload build artifact uses: actions/upload-artifact@v4 with: name: dist-files path: dist/ retention-days: 7

Docker builds

Many companies package their apps into Docker containers for consistent deployments. Here's how to build and push a Docker image in CI:

.github/workflows/docker.yml
jobs: build-and-push: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 # Log in to Docker Hub using secrets - name: Log in to Docker Hub uses: docker/login-action@v3 with: username: ${{ secrets.DOCKER_USERNAME }} password: ${{ secrets.DOCKER_PASSWORD }} # Build and push the image - name: Build and push Docker image uses: docker/build-push-action@v5 with: context: . push: true tags: myuser/myapp:latest,myuser/myapp:${{ github.sha }}

Deployment ‚Äî Shipping Your Code

Deployment is the final stage ‚Äî getting your built code running in production. The exact steps depend on your infrastructure, but the pattern is always the same: download the artifact, stop the old version, start the new one.

Deploying to different environments

.github/workflows/deploy.yml ‚Äî staging + production
on: push: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm ci && npm test deploy-staging: needs: test runs-on: ubuntu-latest environment: staging # uses staging secrets steps: - uses: actions/checkout@v4 - name: Deploy to staging run: | echo "Deploying to staging..." # Your deploy commands here env: DEPLOY_URL: ${{ secrets.STAGING_URL }} API_KEY: ${{ secrets.STAGING_API_KEY }} deploy-production: needs: deploy-staging runs-on: ubuntu-latest environment: production # can require manual approval! steps: - uses: actions/checkout@v4 - name: Deploy to production run: echo "Deploying to production!"

Secrets & Configuration

Pipelines need API keys, passwords, tokens. Never put these directly in your YAML files or code. They'd be visible to everyone with repo access. Use GitHub's built-in secrets system instead.

Setting up secrets in GitHub

  1. Go to your repository on GitHub
  2. Click Settings ‚Üí Secrets and variables ‚Üí Actions
  3. Click "New repository secret"
  4. Give it a name (e.g., AWS_ACCESS_KEY) and paste the value
  5. Reference it in your workflow as ${{ secrets.AWS_ACCESS_KEY }}

Using secrets in workflows

using secrets safely
steps: - name: Deploy to AWS env: AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }} AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }} run: | aws s3 sync ./dist s3://my-bucket # Environment variables (non-secret config) - name: Build with env vars env: NODE_ENV: production API_URL: https://api.myapp.com run: npm run build
üö® Critical security rules Never print secrets with echo. Never commit .env files with real values. Never store secrets in artifacts. GitHub automatically masks secrets in logs (shows *** instead), but only if you use the secrets context.

Environment-specific configuration

GitHub Actions supports "Environments" (staging, production) ‚Äî each with their own set of secrets and optional protection rules like requiring manual approval before deployment proceeds.

Advanced Pipelines ‚Äî Senior Level

These are the patterns that make pipelines fast, maintainable, and powerful. You'll see these in mature engineering orgs.

Matrix builds ‚Äî test across multiple versions

matrix strategy
jobs: test: runs-on: ubuntu-latest strategy: matrix: node-version: [18, 20, 22] os: [ubuntu-latest, windows-latest] steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node-version }} - run: npm test # This runs 6 jobs in parallel: 3 versions √ó 2 OSes

Reusable workflows ‚Äî DRY pipelines

.github/workflows/reusable-test.yml
# Define a reusable workflow on: workflow_call: # this makes it reusable inputs: node-version: type: string default: '20' jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: ${{ inputs.node-version }} - run: npm ci && npm test
another workflow calling the reusable one
jobs: run-tests: uses: ./.github/workflows/reusable-test.yml with: node-version: '22'

Caching ‚Äî making pipelines fast

caching dependencies
steps: # Cache node_modules based on package-lock.json hash - name: Cache dependencies uses: actions/cache@v4 with: path: ~/.npm key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }} restore-keys: | ${{ runner.os }}-node- # With cache hit, npm ci runs in seconds instead of minutes - run: npm ci

Conditional steps and jobs

conditional execution
steps: # Only run on pull requests - name: Comment on PR if: github.event_name == 'pull_request' run: echo "This is a PR!" # Only run if previous step failed - name: Notify failure if: failure() run: echo "Something broke!" # Only deploy on pushes to main - name: Deploy if: github.ref == 'refs/heads/main' && github.event_name == 'push' run: ./deploy.sh

Senior Engineer Patterns

These are the habits and patterns that distinguish senior engineers. Understanding them helps you participate in technical discussions and know where to grow.

Git habits of senior engineers

‚úÇÔ∏è

Small, atomic commits

Each commit does one thing. Easy to understand, easy to revert, easy to cherry-pick.

üßπ

Clean up before PRs

Interactive rebase to squash "wip" commits, rename vague messages, remove dead ends.

üìù

Descriptive PR descriptions

Explain the why, what changed, how to test, and any tradeoffs. Screenshots for UI changes.

üîç

Review their own diffs

Before requesting review, open the PR and read every line. Catch your own mistakes first.

CI/CD principles senior engineers live by

‚ö°

Fast feedback loops

Pipelines should fail fast. Run cheap tests first. A 2-minute feedback loop beats a 20-minute one.

üîí

Protected main branch

Require passing CI before merging. Require at least one code review. Never force-push to main.

üìä

Observability

Track deployment frequency, failure rate, and mean time to recovery. These metrics tell you if CI/CD is working.

üè≥Ô∏è

Feature flags

Deploy dark (hidden) features to production but activate them separately. Enables true continuous deployment without risk.

Conversations you'll now understand

üó£ "The pipeline is red" CI tests are failing. No one can merge until it's fixed. Check the Actions tab to see which step failed.
üó£ "Rebase onto main before you push" Your branch is behind ‚Äî pull the latest main commits under yours so there are no conflicts in the PR.
üó£ "Squash your commits" You have 15 "wip" commits. Use interactive rebase to combine them into 2-3 meaningful commits before the PR.
üó£ "Let's add a gate on that deploy" Add a manual approval step before deploying to production so someone has to click "approve" in GitHub.
üó£ "That should be in a secret, not hardcoded" You put an API key directly in the workflow YAML. Move it to GitHub Secrets and reference it with ${{ secrets.KEY }}.
