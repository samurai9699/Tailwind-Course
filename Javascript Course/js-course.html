üß† Before Day 1: The Developer Mindset

The reason you freeze when building alone isn't that you don't know enough. It's that you don't have a process for when things break.

üî• The Stuck Loop ‚Äî Your Debugging Protocol

  1. State the problem out loud. Write: "I am trying to ___ and it is failing because ___." Vagueness is where bugs hide.
  2. Read the error message ‚Äî actually read it. File name, line number, error type. The answer is almost always there.
  3. Console.log the thing you're unsure about. Not what you think it is ‚Äî what it actually is at runtime.
  4. Rubber duck it. Explain your code line by line as if to a 10-year-old. The bug usually surfaces here.
  5. Binary search your own code. Comment out half. Does it work? Bug is in the removed half. Narrow it down.
  6. Google the exact error + "JavaScript". Read the answer. Understand why it works before you copy it.
  7. Walk away for 10 minutes after 30 minutes stuck. Your brain pattern-matches in the background.

üí° Build Before You're Ready

  1. Read the concept from this course ‚Äî 10 minutes max.
  2. Close the course. Open a blank file. Write the code from memory. It will be wrong. That's the entire point.
  3. Use the error messages to guide you, not the docs. The error tells you exactly what's broken.
  4. Get one tiny thing working. Ugly is fine. Move on from there.
  5. Add one feature. Break it. Fix it. This cycle is the actual job.
üìå Daily Ritual ‚Äî Non-Negotiable

The 5-Minute Dev Log

After every coding session, write in a text file: What did I build? What broke and how did I fix it? What am I still confused about? What will I build tomorrow to reinforce this?

This log becomes the source material for every "describe a challenge you faced" interview question. It compounds.

üìñ Taught Here ‚Äî Concept First, Then Build

1. const and let ‚Äî Never use var again

var is function-scoped and gets hoisted to the top of its function. This causes subtle bugs because it leaks out of blocks like if and for. let and const are block-scoped ‚Äî they only exist inside the {} they were declared in. Rule: use const always. Switch to let only when you know you'll reassign.

// var leaks out of blocks ‚Äî causes real bugs
for (var i = 0; i < 3; i++) {}
console.log(i)  // 3 ‚Äî escaped the for block!

// let/const stay inside their block
for (let j = 0; j < 3; j++) {}
console.log(j)  // ReferenceError: j is not defined

const name = "Alex"
name = "Bob"    // TypeError: Assignment to constant variable

let score = 0
score += 10     // fine ‚Äî let can be reassigned

2. Arrow Functions

Shorter syntax. When the body is a single expression you can drop the return keyword and the braces entirely. They also don't bind their own this ‚Äî which you'll see matters on Day 3.

// Traditional function
function double(n) { return n * 2 }

// Arrow function ‚Äî same thing
const double = (n) => { return n * 2 }

// Implicit return ‚Äî skip return + braces for a single expression
const double = n => n * 2

// Returning an object? Wrap it in () so JS doesn't confuse {} with a block
const makeUser = name => ({ name, active: true })

3. Template Literals

Backtick strings. Embed any JavaScript expression with ${}. Supports multi-line strings natively.

const user = "Sam", score = 42

// Old way ‚Äî fragile, easy to misplace quotes/spaces
const msg = "Hello " + user + ", score: " + score

// Template literal ‚Äî any expression inside ${}
const msg = `Hello ${user}, score: ${score}`
const msg = `Result: ${score > 40 ? "Pass" : "Fail"}`
const html = `
  <div>
    <h2>${user}</h2>
    <p>Score: ${score}</p>
  </div>
`

4. Destructuring

Pull values out of objects and arrays without repetitive dot notation. You'll see this in every real codebase ‚Äî especially with API responses and function parameters.

const user = { name: "Alex", age: 28, address: { city: "Berlin" } }

// Object destructuring
const { name, age } = user                // name="Alex", age=28

// Rename on destructure (new name on the right of the colon)
const { name: userName } = user           // userName="Alex"

// Default value ‚Äî used only if the property is undefined
const { role = "guest" } = user          // "guest" ‚Äî user has no role

// Nested ‚Äî dig into nested objects
const { address: { city } } = user        // city="Berlin"

// Array destructuring ‚Äî position-based
const [first, , third, ...rest] = [10, 20, 30, 40]
// first=10, third=30, rest=[40]  (the , , skips an item)

// In function parameters ‚Äî very common in React and modern JS
function greet({ name, role = "guest" }) {
  return `Hello ${name}, role: ${role}`
}

5. Spread and Rest

The ... operator. As spread it expands an iterable. As rest it collects remaining items. Same symbol, context determines meaning.

// SPREAD ‚Äî expand into individual items
const a = [1, 2], b = [3, 4]
const all = [...a, ...b]                  // [1,2,3,4]

// Spread to copy + override an object (original unchanged)
const defaults = { theme: "dark", lang: "en" }
const prefs    = { ...defaults, lang: "fr" }
// { theme:"dark", lang:"fr" } ‚Äî later values win

// REST ‚Äî collect the remaining arguments into an array
function sum(...nums) {
  return nums.reduce((acc, n) => acc + n, 0)
}
sum(1, 2, 3, 4)  // 10 ‚Äî nums = [1,2,3,4]

Array Methods & Data Transformation

map, filter, reduce, find ‚Äî transforming data is 70% of frontend work

BuildDebugInterview
üìñ Taught Here ‚Äî Concept First, Then Build

Why these methods exist

You constantly receive arrays from APIs and need to transform them. These methods handle 90% of that. They are all pure ‚Äî they return a new array without modifying the original. Exception: sort() mutates in place.

map() ‚Äî Transform every item into something new

const prices = [10, 25, 50]
const withTax = prices.map(p => p * 1.2)   // [12, 30, 60] ‚Äî prices unchanged

// Real use: reshape API objects to what your UI needs
const users = [{ id: 1, first_name: "Alex" }, { id: 2, first_name: "Sam" }]
const names  = users.map(u => u.first_name)   // ["Alex", "Sam"]
const opts   = users.map(u => ({ value: u.id, label: u.first_name }))

filter() ‚Äî Keep only items that pass a test

const products = [
  { name: "Laptop", price: 999, inStock: true  },
  { name: "Desk",   price: 349, inStock: false },
  { name: "Chair",  price: 199, inStock: true  },
]
const available = products.filter(p => p.inStock)
const goodDeal  = products.filter(p => p.inStock && p.price < 500)

reduce() ‚Äî Collapse an array to a single value

The accumulator starts as your initial value (second argument). Whatever you return becomes the next accumulator. This is the most powerful and most misunderstood method.

// Sum ‚Äî start acc at 0
const total = [10, 20, 30].reduce((acc, n) => acc + n, 0)  // 60

// Group by property ‚Äî start acc as empty object
const items = [
  { cat: "tech", name: "Laptop" },
  { cat: "home", name: "Lamp"   },
  { cat: "tech", name: "Phone"  },
]
const grouped = items.reduce((acc, item) => {
  if (!acc[item.cat]) acc[item.cat] = []
  acc[item.cat].push(item.name)
  return acc
}, {})
// { tech: ["Laptop","Phone"], home: ["Lamp"] }

find() and findIndex()

const users = [{ id:1, name:"Alex" }, { id:2, name:"Sam" }]
const user  = users.find(u => u.id === 2)       // { id:2, name:"Sam" }
const index = users.findIndex(u => u.id === 2)  // 1
// find returns undefined if nothing matches ‚Äî always check before using
if (user) console.log(user.name)

sort() ‚Äî the trap

// ‚ö†Ô∏è sort() mutates the original array
// ‚ö†Ô∏è Without a comparator it sorts as strings, not numbers
[10, 2, 1, 20].sort()             // [1,10,2,20] ‚Äî WRONG
[10, 2, 1, 20].sort((a,b) => a-b)  // [1,2,10,20] ‚Äî correct

// Chain them together for powerful one-liners
const result = products
  .filter(p => p.inStock)
  .map(p => p.name)
  .sort((a,b) => a.localeCompare(b))

DOM Manipulation & Events

Query, create, modify, delete ‚Äî making things happen in the browser

Project 1BuildDebug
üìñ Taught Here ‚Äî Concept First, Then Build

The DOM is a JavaScript object tree

When the browser parses HTML it creates a tree of JavaScript objects ‚Äî one per element. Every tag becomes a node you can read, modify, create, or remove. It's just objects and methods.

Selecting elements

// querySelector ‚Äî first match or null
const btn   = document.querySelector('#submit')      // by ID
const input = document.querySelector('.search-box')  // by class
const el    = document.querySelector('nav h1')       // CSS selector

// querySelectorAll ‚Äî all matches as a NodeList
// Convert to array first to use .map()/.filter()
const items = [...document.querySelectorAll('.item')]

Creating and inserting elements

// Pattern: Create ‚Üí Configure ‚Üí Attach
const li = document.createElement('li')
li.textContent = 'Buy milk'   // SAFE ‚Äî auto-escapes <script> etc.
// li.innerHTML = userText    // DANGEROUS if text is user input (XSS)
li.classList.add('task')
li.dataset.id = '42'         // sets data-id="42"
document.querySelector('.list').appendChild(li)

Modifying elements

// Classes
el.classList.add('active')
el.classList.remove('active')
el.classList.toggle('active')    // add if absent, remove if present
el.classList.contains('active') // true or false

// data-* attributes
el.dataset.id       // reads data-id
el.dataset.userId   // reads data-user-id (auto-camelCased)

el.remove()          // removes element from DOM

Event delegation ‚Äî the most important pattern here

Don't attach listeners to every child. Attach one listener to the parent. When a child is clicked, the event bubbles up to the parent. Works for dynamically added children too ‚Äî because the parent already exists.

// ‚ùå 100 items = 100 listeners, breaks for new items added later
document.querySelectorAll('.delete-btn')
  .forEach(btn => btn.addEventListener('click', handleDelete))

// ‚úÖ One listener on parent ‚Äî works for any number of children, including future ones
document.querySelector('.task-list').addEventListener('click', (e) => {
  // e.target = the exact element that was clicked
  if (e.target.matches('.delete-btn')) {
    const id = e.target.closest('li').dataset.id  // walk up to the li
    deleteTask(id)
  }
  if (e.target.matches('.task-text')) {
    toggleTask(e.target.closest('li').dataset.id)
  }
})

// e.preventDefault() ‚Äî stops the browser's default action
form.addEventListener('submit', e => { e.preventDefault(); handleSubmit() })

Async JavaScript ‚Äî Callbacks ‚Üí Promises ‚Üí Async/Await

The mental model that unlocks everything once it clicks

BuildDebugInterview
üìñ Taught Here ‚Äî Concept First, Then Build

The problem: JavaScript is single-threaded

JS processes one thing at a time. If it blocked waiting for a network request, your entire page would freeze. The solution: schedule work for later and continue running. When the work finishes, come back to it.

The Event Loop ‚Äî how "later" actually works

// Synchronous code runs immediately on the call stack
// Callbacks/promises go into a queue, processed only when the stack empties

console.log('1')
setTimeout(() => console.log('2'), 0)   // queued even with 0ms delay
console.log('3')
// Output: 1, 3, 2 ‚Äî the stack runs to completion first

Callbacks ‚Äî the original pattern (and why they break down)

// Callback hell ‚Äî each step nested inside the previous one
getUser(id, (user) => {
  getOrders(user.id, (orders) => {
    getProduct(orders[0].productId, (product) => {
      // Error handling is a nightmare here
      console.log(product)
    })
  })
})

Promises ‚Äî a model for future values

A Promise represents a value that doesn't exist yet. States: pending, fulfilled, rejected. You rarely create them manually ‚Äî APIs return them. But knowing how they're built helps you understand them.

// Create a Promise
function delay(ms) {
  return new Promise((resolve, reject) => {
    setTimeout(resolve, ms)  // call resolve() to fulfill, reject() to fail
  })
}

// Consume with .then() and .catch()
delay(1000)
  .then(() => fetchUser(1))
  .then(user => console.log(user))
  .catch(err => console.error(err))

Async/Await ‚Äî same thing, readable syntax

Not a new system. Same Promises, same event loop. async makes a function always return a Promise. await pauses that function until the Promise resolves. Reads like synchronous code.

async function loadOrder(userId) {
  try {
    const user    = await getUser(userId)      // pauses here until resolved
    const orders  = await getOrders(user.id)
    const product = await getProduct(orders[0].productId)
    return product
  } catch (err) {
    // catches ANY failure from any await in the try block
    console.error('Failed:', err)
    throw err   // re-throw so the caller knows
  }
}

// ‚ö†Ô∏è async functions ALWAYS return a Promise, even if you return a plain value
async function getName() { return "Alex" }
getName()        // Promise{"Alex"} ‚Äî NOT "Alex"
await getName()  // "Alex"

// Run operations in parallel instead of sequentially
const [user, posts] = await Promise.all([fetchUser(id), fetchPosts(id)])
// Both fire at once. Resolves when both complete. Rejects if either fails.
// Use Promise.allSettled to handle partial failures gracefully.

How an HTTP request works

Your browser sends a request with a method (GET, POST, etc.) and a URL. The server responds with a status code (200 = OK, 404 = not found, 500 = server error) and a body (usually JSON). fetch() returns a Promise resolving to a Response ‚Äî you then call .json() on it (also async) to get the actual data.

The correct fetch pattern ‚Äî commit this to memory

async function fetchData(url) {
  try {
    const res = await fetch(url)

    // ‚ö†Ô∏è fetch only throws on NETWORK failure (no internet, DNS failure)
    // For 404, 500, etc. ‚Äî res.ok is false but NO exception is thrown
    // You MUST check res.ok manually. res.ok is true for 200-299.
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}: ${res.statusText}`)
    }

    const data = await res.json()
    return data

  } catch (err) {
    console.error('Fetch failed:', err.message)
    throw err   // re-throw so caller can handle it in the UI
  }
}

Sending data with POST/PATCH

const res = await fetch('/api/recipes', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' }, // required
  body: JSON.stringify({ title: 'Pasta', ingredients: ['flour'] })
})

UI states you must always handle

async function loadData() {
  container.innerHTML = '<p>Loading...</p>'   // 1. Loading

  try {
    const data = await fetchData('/api/items')

    if (data.length === 0) {                     // 2. Empty
      container.innerHTML = '<p>Nothing here yet.</p>'
      return
    }

    render(data)                                  // 3. Success

  } catch (err) {                                 // 4. Error
    container.innerHTML = `<p class="error">Failed: ${err.message}</p>`
  }
}

// Debounce ‚Äî wait until user stops typing before firing the request
function debounce(fn, delay) {
  let timer
  return (...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }
}
input.addEventListener('input', debounce(e => search(e.target.value), 500))

Why modules matter

When your app is one 500-line file, your UI code, data logic, network calls, and storage are all tangled together. A bug in one place can silently break everything. Modules force single responsibility ‚Äî each file declares exactly what it exports and imports.

Named exports and imports

// utils.js ‚Äî pure functions only (no DOM, no fetch)
export function formatDate(d) {
  return new Date(d).toLocaleDateString('en-US', { month:'short', day:'numeric' })
}
export const capitalize = s => s[0].toUpperCase() + s.slice(1)

// api.js ‚Äî all fetch calls live here, nothing else
import { formatDate } from './utils.js'
export async function fetchRecipes() {
  const res = await fetch('/api/recipes')
  if (!res.ok) throw new Error(res.status)
  const items = await res.json()
  return items.map(r => ({ ...r, dateFormatted: formatDate(r.created_at) }))
}

// main.js ‚Äî event listeners only, wires everything together
import { fetchRecipes } from './api.js'
import { renderList } from './ui.js'
import { save, load } from './storage.js'

localStorage ‚Äî the right way

localStorage only stores strings. Always serialize with JSON.stringify, deserialize with JSON.parse. Always wrap in try/catch ‚Äî it throws in private browsing mode and when storage is full.

// storage.js
export function save(key, value) {
  try {
    localStorage.setItem(key, JSON.stringify(value))
  } catch (err) {
    console.warn('localStorage unavailable:', err)
  }
}

export function load(key, fallback = null) {
  try {
    const raw = localStorage.getItem(key)
    return raw !== null ? JSON.parse(raw) : fallback
  } catch {
    return fallback  // corrupted data ‚Äî return fallback
  }
}

export const remove = key => localStorage.removeItem(key)

The module structure for a frontend app

src/
  utils.js    ‚Üê pure helper functions. No side effects.
  storage.js  ‚Üê localStorage wrappers only
  api.js      ‚Üê all fetch calls. Nothing else.
  ui.js       ‚Üê DOM rendering functions. No fetch, no storage.
  state.js    ‚Üê the app's in-memory data (source of truth)
  main.js     ‚Üê event listeners, wires all modules together

What Express does

Node's built-in http module is very low-level. Express wraps it with clean routing, middleware support, and response helpers. Every web framework in every language (Django, Rails, Laravel) works on the same idea: map URL patterns + HTTP methods to handler functions.

Setup, routes, and parameters

const express = require('express')
const app = express()

// Middleware comes BEFORE routes ‚Äî order matters
app.use(express.json())    // parses JSON request bodies into req.body

// Route: method + path + handler
app.get('/users', (req, res) => {
  res.json(users)           // sends JSON with 200 status
})

// Route parameters ‚Äî :id is a wildcard, accessed via req.params.id
app.get('/users/:id', (req, res) => {
  const id   = parseInt(req.params.id)  // ‚ö†Ô∏è req.params are always strings
  const user = users.find(u => u.id === id)
  if (!user) return res.status(404).json({ error: 'Not found' })
  res.json(user)
})

// Query params ‚Äî /users?role=admin&limit=10 ‚Üí req.query
app.get('/users', (req, res) => {
  const { role, limit = '20' } = req.query
  const filtered = role ? users.filter(u => u.role === role) : users
  res.json(filtered.slice(0, parseInt(limit)))
})

// Creating a resource ‚Äî use 201 Created, not 200
app.post('/users', (req, res) => {
  const { name, email } = req.body
  if (!name || !email) {
    return res.status(400).json({ error: 'name and email required' })
  }
  const user = { id: Date.now(), name, email }
  users.push(user)
  res.status(201).json(user)
})

// Delete ‚Äî use 204 No Content (no body)
app.delete('/users/:id', (req, res) => {
  const index = users.findIndex(u => u.id === +req.params.id)
  if (index === -1) return res.status(404).json({ error: 'Not found' })
  users.splice(index, 1)
  res.status(204).send()
})

app.listen(3000, () => console.log('Running on port 3000'))

Middleware ‚Äî the most important concept in Express

Middleware is any function with (req, res, next). It runs in the order it's defined. It can modify req, send a response (ending the chain), or call next() to pass control forward. If a middleware doesn't call next() and doesn't send a response, the request hangs.

// Logger ‚Äî runs before every route
app.use((req, res, next) => {
  console.log(`${req.method} ${req.path} ‚Äî ${new Date().toISOString()}`)
  next()  // MUST call next() or the request hangs
})

// Validation middleware ‚Äî only for specific routes
function validateRecipe(req, res, next) {
  const { title, ingredients } = req.body
  if (!title || !ingredients) {
    return res.status(400).json({ error: 'title and ingredients required' })
  }
  next()
}
app.post('/recipes', validateRecipe, (req, res) => { /* ... */ })

// CORS ‚Äî allows your frontend (different origin) to call your API
const cors = require('cors')
app.use(cors({ origin: 'http://localhost:5500' }))

// Global error handler ‚Äî 4 params, must be LAST
app.use((err, req, res, next) => {
  console.error(err.stack)
  res.status(err.status || 500).json({ error: err.message })
})

BuildInterview

Why SQL?

In-memory arrays disappear when the server restarts. SQL databases persist to disk and give you structured querying, constraints, and relationships. SQLite is a file-based database ‚Äî no server to install, perfect for learning and small apps.

Defining tables and types

-- SQL uses its own comment syntax
-- CREATE TABLE defines the schema ‚Äî columns, types, constraints

CREATE TABLE users (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,  -- auto ID
  name       TEXT    NOT NULL,                   -- required
  email      TEXT    UNIQUE NOT NULL,            -- unique + required
  created_at TEXT    DEFAULT (datetime('now'))  -- auto timestamp
);

CREATE TABLE recipes (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  title       TEXT    NOT NULL,
  ingredients TEXT    NOT NULL,
  category    TEXT    DEFAULT 'other',
  user_id     INTEGER REFERENCES users(id) ON DELETE CASCADE  -- foreign key
);

CRUD queries

-- SELECT
SELECT * FROM recipes;
SELECT title, category FROM recipes WHERE category = 'italian';
SELECT * FROM recipes ORDER BY title ASC LIMIT 10;

-- INSERT
INSERT INTO recipes (title, ingredients, user_id)
VALUES ('Pasta', '["flour","eggs"]', 1);

-- UPDATE
UPDATE recipes SET title = 'Carbonara' WHERE id = 3;

-- DELETE
DELETE FROM recipes WHERE id = 3;

-- JOIN ‚Äî combine rows from two tables on a matching column
SELECT recipes.title, users.name AS author
FROM recipes
JOIN users ON recipes.user_id = users.id
WHERE recipes.category = 'italian';

Using SQLite in Node with better-sqlite3

better-sqlite3 is synchronous (no callbacks or Promises needed) and very fast. Ideal for learning and production small-scale apps.

const Database = require('better-sqlite3')
const db = new Database('./app.db')

// Create table on startup ‚Äî IF NOT EXISTS prevents errors on re-run
db.exec(`
  CREATE TABLE IF NOT EXISTS recipes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    ingredients TEXT NOT NULL,
    category TEXT DEFAULT 'other'
  )
`)

// ALWAYS use parameterized queries ‚Äî the ? placeholders prevent SQL injection
// NEVER use string interpolation: `WHERE id = ${req.params.id}` ‚Üê DANGEROUS

// .get(id) ‚Äî returns one matching row, or undefined
const recipe = db.prepare('SELECT * FROM recipes WHERE id = ?').get(id)

// .all() ‚Äî returns an array of all matching rows
const recipes = db.prepare('SELECT * FROM recipes WHERE category = ?').all(category)

// .run() ‚Äî for INSERT, UPDATE, DELETE
// returns { changes: 1, lastInsertRowid: 5 }
const result = db
  .prepare('INSERT INTO recipes (title, ingredients) VALUES (?, ?)')
  .run(title, JSON.stringify(ingredients))

const newId = result.lastInsertRowid  // use this to return the created resource

Authentication ‚Äî JWT & Password Hashing

How login actually works: bcrypt, tokens, protected routes

BuildInterview
üìñ Taught Here ‚Äî Concept First, Then Build

Why you never store plain passwords

If your database is breached, plain passwords expose users on every other site where they reuse that password. bcrypt hashes passwords with a one-way algorithm plus a random salt ‚Äî you literally cannot reverse it. Even if two users have the same password, their hashes are different because of the salt.

bcrypt ‚Äî hashing and verifying

const bcrypt = require('bcryptjs')

// REGISTER ‚Äî hash before saving to database
const saltRounds = 12   // higher = slower = harder to brute-force
const hashed = await bcrypt.hash(plaintextPassword, saltRounds)
// "$2a$12$..." ‚Äî a long string. Store this in your DB, never the original.

// LOGIN ‚Äî compare plain input against the stored hash
const isValid = await bcrypt.compare(plaintextPassword, hashedFromDB)
// true or false ‚Äî bcrypt handles the salt internally

JWTs ‚Äî stateless authentication

After login the server creates a signed token containing the user's ID. The client stores it and sends it in every request. The server verifies the signature ‚Äî if valid, the user is authenticated. No session storage needed on the server.

const jwt = require('jsonwebtoken')
const SECRET = process.env.JWT_SECRET  // long random string, in .env

// SIGN ‚Äî create a token after successful login
const token = jwt.sign(
  { userId: user.id, email: user.email },  // payload ‚Äî never include password
  SECRET,
  { expiresIn: '7d' }
)
// "eyJhbGciOiJ..." ‚Äî three base64 parts: header.payload.signature

// VERIFY ‚Äî check a token (done in middleware, not in routes)
try {
  const payload = jwt.verify(token, SECRET)  // throws if invalid or expired
  // payload = { userId: 1, email: "...", iat: ..., exp: ... }
} catch (err) {
  // JsonWebTokenError: invalid signature
  // TokenExpiredError: jwt expired
}

The authentication middleware pattern

// middleware/authenticate.js
function authenticate(req, res, next) {
  const header = req.headers.authorization  // "Bearer eyJ..."
  if (!header?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token' })
  }
  try {
    const payload = jwt.verify(header.slice(7), process.env.JWT_SECRET)
    req.user = payload   // attach user info for downstream handlers
    next()
  } catch {
    res.status(401).json({ error: 'Invalid or expired token' })
  }
}

// Protect routes by adding authenticate before the handler
app.post('/recipes', authenticate, (req, res) => {
  // req.user.userId available here
  const recipe = createRecipe({ ...req.body, userId: req.user.userId })
  res.status(201).json(recipe)
})

Sending the auth token from your frontend

// After login ‚Äî save the token
const { token } = await res.json()
localStorage.setItem('token', token)

// Helper that adds the auth header automatically
async function authFetch(url, options = {}) {
  const token = localStorage.getItem('token')
  return fetch(url, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token && { Authorization: `Bearer ${token}` }),
      ...options.headers,
    },
  })
}
// Replace fetch() with authFetch() everywhere in your frontend

Global error handling on the backend

// A custom error class ‚Äî gives errors a status code and a clean type
class AppError extends Error {
  constructor(message, status = 500) {
    super(message)
    this.status = status
    this.name = 'AppError'
  }
}

// In routes: throw AppError and call next(err)
app.get('/recipes/:id', (req, res, next) => {
  try {
    const recipe = db.prepare('SELECT * FROM recipes WHERE id=?').get(req.params.id)
    if (!recipe) throw new AppError('Recipe not found', 404)
    res.json(recipe)
  } catch (err) { next(err) }   // passes to global error handler
})

// Global error handler ‚Äî 4 params, must be registered LAST
app.use((err, req, res, next) => {
  const status  = err.status || 500
  const message = err.name === 'AppError' ? err.message : 'Internal server error'
  console.error(`[${status}] ${err.message}`)
  res.status(status).json({ error: message })
})

// 404 catch-all ‚Äî routes that don't exist
app.use((req, res) => {
  res.status(404).json({ error: `Route ${req.method} ${req.path} not found` })
})
