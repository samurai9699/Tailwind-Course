Lesson 01

What is Vite & Why Should You Care?

Before touching any code, let's understand what Vite actually is and why the entire JavaScript community switched to it seemingly overnight.

The Old World: Webpack & CRA

For years, Create React App (CRA) was the go-to way to start a React project. It was powered by Webpack under the hood â€” a powerful but slow bundler. Every time you saved a file, Webpack had to re-process your entire application. On large projects, that meant waiting 30â€“60 seconds just to see your change.

âš ï¸

Create React App is now deprecated. The official React docs no longer recommend it. Vite is the modern replacement â€” and it's dramatically better.

The New World: Vite

Vite (French for "fast", pronounced like "veet") was created by Evan You â€” the same person who built Vue.js. It takes a completely different approach:

  • In development: Vite serves your source files directly using native ES Modules in the browser. No bundling needed. The browser does the importing itself.
  • In production: Vite bundles with Rollup â€” a highly optimized bundler â€” producing tight, fast output.

How much faster is it, really?

ToolCold StartHot ReloadBuild
Webpack / CRA 20â€“60+ seconds 2â€“10 seconds Slow
Vite < 1 second Instant (HMR) Fast (Rollup)

Vite under the hood (simplified)

Vite splits your code into two categories:

  • Dependencies (things in node_modules like React itself): Pre-bundled once using esbuild, which is written in Go and 10â€“100Ã— faster than JavaScript-based bundlers.
  • Your source code (components, pages, utils): Served as-is over native ESM. Vite transforms files on demand, only when the browser requests them.
ğŸ’¡

Think of Vite like a smart chef who preps your pantry items once (dependencies), then cooks your actual meal (your code) to order â€” rather than re-prepping everything every time you change a single ingredient.

Lesson 04

Dev Server & Hot Module Replacement (HMR)

HMR is one of Vite's killer features. It updates only the changed module in your browser without doing a full page refresh â€” keeping your app state intact.

How HMR Works in Vite

When you save a file:

  1. Vite detects the change via file system watchers.
  2. It re-transforms only the changed module and its direct dependents.
  3. It sends an update to the browser over a WebSocket connection.
  4. The browser swaps out only the changed module â€” your component state stays alive.
ğŸ’¡

HMR means if you have a form with typed text and you update the form's CSS, your text stays there. No refresh, no lost state. This is a massive productivity win.

Configuring the Dev Server

You can customize the dev server in vite.config.js:

js â€” vite.config.js
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    port: 3000,        // change the default port
    open: true,         // auto-open browser on start
    host: true,         // expose to network (for mobile testing)
    strictPort: true,   // fail if port is taken (don't increment)
  }
})

Proxying API Requests

One of the most important dev server features: proxying API calls to avoid CORS issues during development.

js â€” vite.config.js
export default defineConfig({
  server: {
    proxy: {
      // Requests to /api/... will be forwarded to your backend
      '/api': {
        target: 'http://localhost:8000',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
})

Now fetch('/api/users') in your React code will actually hit http://localhost:8000/users â€” no CORS errors, no backend config needed.

Environment Variables

Environment variables let you store configuration that changes between environments â€” like API URLs, API keys, and feature flags â€” without hardcoding them in your source code.

How Vite Handles Env Variables

Vite uses .env files. The key rule: only variables prefixed with VITE_ are exposed to your client-side code. This prevents accidentally leaking server secrets.

.env files (precedence order)
.env                  # loaded always
.env.local            # loaded always, git-ignored
.env.development      # loaded when NODE_ENV=development
.env.production       # loaded when NODE_ENV=production
.env.development.local # local overrides for dev
.env
# âœ… Exposed to client (VITE_ prefix)
VITE_API_URL=https://api.example.com
VITE_APP_TITLE=My Awesome App
VITE_FEATURE_DARK_MODE=true

# âŒ NOT exposed to client (no VITE_ prefix)
DATABASE_PASSWORD=supersecret123
SECRET_KEY=abc123

Accessing Variables in Your Code

jsx
// Access via import.meta.env
const apiUrl = import.meta.env.VITE_API_URL
const appTitle = import.meta.env.VITE_APP_TITLE

// Built-in variables
const isDev = import.meta.env.DEV       // true in dev
const isProd = import.meta.env.PROD     // true in prod
const mode = import.meta.env.MODE       // "development" or "production"

function App() {
  return (
    <div>
      <h1>{import.meta.env.VITE_APP_TITLE}</h1>
      {import.meta.env.DEV && <p>ğŸ”§ Dev Mode</p>}
    </div>
  )
}
ğŸ”’

Never put secrets (database passwords, private API keys) with the VITE_ prefix. They will be bundled into your JavaScript and visible to anyone who views source. Only public-facing values belong in VITE_ variables.

Writing Your Own Plugin

Plugins are just objects with lifecycle hooks. Here's a simple one that adds a build timestamp:

js â€” custom plugin
// A plugin that injects build time into your app

    name: 'build-timestamp',
    transformIndexHtml(html) {

  }
